# V5 Adoption Friction Correction Train Design

**Date:** 2026-07-22 **Status:** authored for autonomous implementation from the owner's resolve-open-issues directive; publication requires owner confirmation **Author:** Claude (Fable 5) with Chris Purcell / L3DigitalNet

## Problem and goal

Project Standards 5.7.0 leaves six consumer-facing defects, filed as GitHub issues #26–#31 on 2026-07-22 from real consumer repositories completing or attempting V4 → V5 migrations.

- **#31** — Python Tooling 1.7 renders `[tool.pytest.ini_options].testpaths` as a managed key whose value is the hardcoded literal `["tests"]` with `governing_options = []`. A repository whose test suite deliberately lives elsewhere cannot adopt the package at all: `CP-CONSUMER-CONFLICT` reports `governing options: none declared`, and neither offered resolution (relocate the suite, or a consumer-ownership escape that does not exist) is available.
- **#28** — markdown-frontmatter 1.4 declares no ownership option for `.github/workflows/validate-standards.yml`, so a consumer whose caller workflow carries `paths:` trigger filters and rationale comments blocks on `CP-MIGRATION-LEGACY-DIGEST` whose hint ("preserve the local version explicitly") advertises a resolution the package cannot provide. The only workaround is restoring shipped bytes, discarding the customization.
- **#27** — a consumer `.markdownlint.json` written at adoption time in 2026-06 holds bytes that are parsed-JSON-equal to every shipped form but serialize CJK punctuation as literal UTF-8 instead of `\uXXXX` escapes. Investigation (see Evidence) proves no release ever shipped the literal form; the consumer file (digest `sha256:4c1c089d0552a6118f6a8b7d85bae1bd762da41d601d1c489bdb9143f6a2d548`) is a consumer-side re-serialization from the pre-`adopt`-CLI era. It matches no declared legacy signature, so migration hard-blocks with three findings reporting only opaque digests, and nothing distinguishes "customized rule" from "identical file, different escape form".
- **#26** — `format-frontmatter` unconditionally single-quotes every frontmatter scalar, doubling apostrophes (`'Apple''s'`), while the markdown-tooling Prettier configuration (`singleQuote: true` for `**/*.md`) prefers the quote style requiring no escapes and rewrites the same scalar to `"Apple's"`. Each tool's output is the other's violation; the two companion packages have no fixed point for any frontmatter string containing an apostrophe.
- **#30** — `emit_legacy_authority_warning()` fires from the shared `_resolve_state()` path for every resolution-routed command in `LEGACY_ONLY` state, including the `agent-handoff size-report` and `shape-check` runs that `UPGRADING.md` §2 explicitly directs consumers to perform before migration apply. The tool warns "migrate before using the V5 control plane" at exactly the moment the documentation says to run it.
- **#29** — reported as "`format-frontmatter FILE` with no mode flag silently does nothing and exits 0". The symptom does not reproduce against 5.7.0: the bare invocation runs check mode (as `docs/usage.md` documents) in a synthetic consumer and in both originating repositories. Investigation instead confirmed three adjacent defects that produce or entrench the confusion: `--check` is parsed but never read (a dead flag), an explicitly named file matching a config `exclude` pattern or the frontmatter denylist is silently dropped (the one code path yielding the reported silent-exit-0 shape, contradicting the collector's own named-file rationale), and `--help` does not state the default mode.

The goal is to make the pytest collection roots declarable through a closed option, give the customized validate-standards caller a declared preservation path, accept the proven legacy markdownlint byte form, converge the two formatters on one frontmatter serialization, scope the legacy-authority message to what is actually happening, and make `format-frontmatter`'s mode and path handling explicit — while preserving fail-closed ownership, immutable released payloads, exact-version validation outcomes, and additive schema evolution. The corrections ship together as Project Standards 5.8.0 and close GitHub issues #26–#31 only after publication evidence is available.

## Evidence established during investigation

- `testpaths` is hardcoded in `standards/python-tooling/versions/1.7/providers/python_tooling.py:195` and declared as key-scoped contribution `pytest-testpaths` with `governing_options = []` (`payload.toml:280-287`); `[tool.pytest.ini_options]` is deliberately key-granular so undeclared keys (`pythonpath`, `extraPaths`) stay consumer-owned.
- No released tag ever shipped `.markdownlint.json` with literal CJK bytes: every shipped form from v2.0.0 through markdown-tooling 1.7 is the escaped form (`sha256:51204b51…`), verified per tag; a pickaxe for literal fullwidth punctuation over all history returns nothing. The blocked consumer file's exact bytes were recovered from the originating repository's adoption commit (2026-06-07, pre-dating the `adopt` CLI), digest `sha256:4c1c089d…`, parsed-JSON-equal to the shipped resource. The correction mechanism mirrors the issue #10 fix (commit `0c7ac34`): append the proven digest to the whole-file signature's multi-valued `known_content_digests`.
- The relinquishment gate (`control_plane/migration.py:1185-1209`) only clears whole-file signatures whose target is not materialized; markdown-frontmatter composes four YAML units into `validate-standards.yml` (`workflow-name`, `workflow-on` — both `create-only` — `workflow-permissions`, `workflow-frontmatter-job`), and no other package targets that file, so an ownership option must gate those contributions off for the relinquishment path to validate. The payload model additionally rejects `consumer_owned_intent_pointer` and `unknown_content_disposition` on the same signature as mutually exclusive (`package_contract/payload.py:645-650`), so a signature offers either intent-pointer relinquishment or unknown-content preservation, never both.
- Python Tooling manages a fourth consumer of the collection roots: `vscode-pytest-args` renders `python.testing.pytestArgs = ["tests"]` into `.vscode/settings.json` (`python_tooling.py:447`, `payload.toml:499-505`).
- `normalize_lists` (`format_frontmatter.py:308-358`) decodes every uncommented block-list item with `yaml.BaseLoader` and re-emits it through the quote emitter, discarding the item's original spelling — so list items do not share the scalar-line verbatim bypass and the acceptance-preservation rule must be stated for them explicitly.
- `format-frontmatter`'s quoting is the hand-rolled `_emit_single_quoted` (`format_frontmatter.py:157-159`) applied type-blind to the author's raw text; the single-quote choice is a visual-consistency convention (2026-06-08 frontmatter-suite design), not a semantic requirement, and the linting-formatting design assumed Prettier would not touch frontmatter quoting — the assumption the observed fight violates. Critically, `_requote_scalar_line` already preserves any input that is single-quoted verbatim ("already single-quoted -> idempotent"), so the fight enters only through double-quoted and unquoted input being force-converted; widening what the emitter preserves can therefore never fail a previously-passing document. A latent corruption also lives here: a double-quoted scalar whose decoded value contains control characters (e.g. `"a\nb"`) is re-emitted as literal control bytes inside single quotes, altering the document structure.
- `_source_roots` (`python_tooling.py:82-99`) hardcodes `tests` into the checker `include` and Ruff `src` lists for both layouts, independent of the pytest collection roots — so declaring an alternate collection root without also feeding the checker surface would collect tests the strict gate never type-checks.
- The legacy-authority warning text lives in one function (`emit_legacy_authority_warning`, `control_plane/command_resolution.py:85-95`) with a process-wide once-flag and three call sites: the shared `_resolve_state` resolver (`command_resolution.py:229`), the legacy-only validation path (`control_plane/cli.py:837`), and the frontmatter compatibility wrapper (`validate_frontmatter.py:881`). Rewording the one function corrects every route; tests must cover all three.
- `meta/versioning.md` makes the previously-passing rule absolute: any change that can turn a previously-passing consumer document into a failure is MAJOR without exception, while additive leniency (fail-to-pass) is MINOR-safe. Every behavior change in this train must therefore only widen what passes.
- Issue #29's reproduction transcript could not be reproduced with the released 5.7.0 tool in either originating repository; the silent-drop path for explicitly named excluded files (`validate_frontmatter.py:485`) and the never-read `--check` flag (`format_frontmatter.py:655` vs `:700`) are confirmed real.

## Scope

In scope:

- author Python Tooling 1.8 adding a `pytest.test_paths` option governing the rendered `testpaths` value, defaulting to `["tests"]`, with 1.7 → 1.8 and V4 legacy migrations;
- author markdown-tooling 1.8 whose `legacy-markdownlint-config` signature also accepts the proven literal-UTF-8 byte form, with 1.7 → 1.8 succession and legacy migration re-pointed;
- author markdown-frontmatter 1.5 adding a `workflow_ownership` option that relinquishes the `validate-standards.yml` composition through the established intent-pointer path, and documenting the converged frontmatter serialization;
- change `format-frontmatter` scalar emission to the minimal-escape quote rule — widening acceptance only — so a repository state exists that both the frontmatter check and the markdown-tooling Prettier gate accept;
- state `format-frontmatter`'s default mode in `--help`, emit a diagnostic when an explicitly named file is skipped by config exclusion or the frontmatter denylist, and resolve the dead `--check` flag through an internal mode-resolution unit;
- reword the legacy-authority warning into a factual note that does not contradict the documented pre-migration workflow;
- extend the whole-file `CP-CONSUMER-CONFLICT` rendering with a first-differing-line pointer so byte-form mismatches are self-diagnosable;
- retain every predecessor package version while advancing the Catalog 5 defaults to python-tooling 1.8, markdown-tooling 1.8, and markdown-frontmatter 1.5;
- publish and verify Project Standards 5.8.0, then close #26–#31 with release evidence.

Out of scope:

- changing any released payload bytes (python-tooling 1.7, markdown-tooling 1.7, markdown-frontmatter 1.4, and all predecessors stay byte-identical);
- changing the markdown-tooling Prettier configuration or the shipped `.markdownlint.json` escape form (the shipped form is stable; the legacy form is accepted, not adopted);
- a general parsed-JSON comparison mode for whole-file targets (digest identity remains the integrity primitive; the known-digest lineage is the escape hatch);
- automatic retirement of arbitrary consumer jobs in `validate-standards.yml` beyond the declared legacy signature handling;
- changing `format-frontmatter`'s type-blind raw-text principle, key ordering, list normalization, or any non-quote formatting behavior;
- suppressing the legacy-authority note entirely or adding per-command warning matrices;
- adding new runtime dependencies or test frameworks.

## Approved approach

### Python Tooling 1.8 pytest collection roots

Python Tooling 1.8 copies the complete 1.7 payload and changes only its config schema, provider, migrations, documentation, and integrity metadata. The `pytest` object gains `test_paths`: an array of the same constrained safe-relative-path strings as `additional_source_roots` string entries, `uniqueItems`, `minItems 1`, default `["tests"]`. `_pytest_key` renders `testpaths` from the resolved option instead of the hardcoded literal; `_source_roots` replaces its hardcoded `tests` entry with the declared collection roots so the checker `include` and Ruff `src` always cover what pytest collects while collection roots never enter `coverage.run.source` on their own; and the VS Code settings render `python.testing.pytestArgs` from the same list, keeping editor discovery aligned with CLI collection. Cross-option overlap is legal and pinned: a path in both options dedupes first-wins in `include` and keeps its `additional_source_roots` coverage semantics. The `pytest-testpaths`, checker-configuration, Ruff-configuration, and `vscode-pytest-args` contributions all declare `/pytest/test_paths` among their governing options. Undeclared configurations render byte-identical 1.7 output, keeping reconciliation quiet for every existing consumer. The key-granular pytest table ownership is unchanged: `pythonpath` and other undeclared keys remain consumer-owned.

The 1.8 migration set wires `python-tooling-1-7-to-1-8` and `legacy-v4-to-1-8` carrying option values verbatim; the 1.7 → 1.8 affected-contribution list enumerates the actual payload delta — `pytest-testpaths`, `vscode-pytest-args`, plus every checker and Ruff contribution whose rendered value now consumes the collection roots — and a reconstruction assertion proves the list covers the delta.

### markdown-tooling 1.8 legacy byte-form acceptance

markdown-tooling 1.8 copies the complete 1.7 payload — all resources byte-identical — and changes only the payload manifest, migrations, documentation, and integrity metadata. The `legacy-markdownlint-config` signature's `known_content_digests` gains the proven literal-form digest alongside the shipped escaped digest, exactly as `legacy-editorconfig` carries the revised and released lineages after the issue #10 correction. A consumer holding the literal form migrates to managed ownership and receives the current shipped escaped bytes; behavior is identical because the two forms are parsed-JSON-equal. Because no release shipped these bytes, they are pinned under an observed-consumer fixture namespace (separate from `tests/fixtures/legacy_releases/`, whose meaning stays "exact released artifacts") with sanitized provenance — recovery date, originating adoption commit date, and asserted digest, no repository identifier — preserving the released-versus-observed distinction the issue #9/#10 correction established. The migration is re-pointed (`markdown-tooling-1-7-to-1-8`, legacy → 1.8).

### markdown-frontmatter 1.5 caller ownership and preservation

markdown-frontmatter 1.5 copies the 1.4 payload and adds `workflow_ownership` (`managed` \| `consumer-owned`, default `managed`) to the config schema, mirroring the markdown-tooling caller-ownership pattern from the issue #12 correction. The four `validate-standards.yml` contributions gain `when_any` gates on `workflow_ownership = "managed"`, so `consumer-owned` leaves nothing materializing that target and the whole-file relinquishment gate validates against the signature's new `consumer_owned_intent_pointer`. The payload model forbids combining that pointer with unknown-content preservation on one signature, so the signature offers exactly the intent-pointer path, and the 1.5 migration provider branches as markdown-tooling's callers do: valid consumer-owned intent → relinquishment claim, customized caller preserved byte-for-byte; known digest → today's migration; unknown digest without declared intent → the existing block, now with an escape the consumer can actually follow. Consumers who want the package's caller keep the default; consumers who own their caller declare it in the legacy YAML before migration (satisfying the UPGRADING.md §3 instruction that is currently unfollowable for this package) or in the V5 configuration afterward, accepting that the caller file — including its job wiring — becomes wholly consumer-maintained, with the documented path back to `managed` (restore the shipped caller, flip the option). The claim matrix — missing, malformed, unknown-with-intent, unknown-without-intent, known, and consumer-owned states — is verified through full preview, apply, and convergence tests.

### Frontmatter serialization convergence

`_emit_single_quoted` is replaced by a minimal-escape emitter implementing the same preference Prettier applies under `singleQuote: true`, constrained by the previously-passing rule so acceptance only widens. Already-single-quoted input keeps its existing verbatim bypass — every 5.7.0-canonical document stays green forever. Double-quoted input is preserved when double-quoting is the minimal style for the decoded value (per the FR-008 cost model) or the value contains control characters only double-quoted escapes can represent — which also fixes the latent corruption where such values were re-emitted as literal control bytes — and is converted to single quotes otherwise, exactly as today. Unquoted input is quoted in the minimal style of its raw text. Block-list items get the same acceptance-preservation rule stated explicitly, because `normalize_lists` reconstructs items rather than editing lines: an item whose current spelling is either accepted form keeps that spelling through the structural normalization (block style, first-wins dedupe, indent), and only items in neither form are re-spelled minimally. The type-blind raw-text principle is untouched — dates, `on`/`off`, and identifier-like numbers keep exact characters. Parity is enforced by tests that run the pinned Prettier from the repository's lockfile over the emission corpus and assert byte-stability in both directions, with the oracle authoritative on divergence, so the rule cannot drift from the formatter it must agree with. All emission call sites (requote, list normalization, scaffolding, doc-type inference, `--bump-updated`) route through the one emitter.

The converged resting state is Prettier's: `"Apple's"` satisfies both tools, legacy `'Apple''s'` remains accepted by the frontmatter check, and Prettier's one-time rewrite of the legacy spelling lands in a state both tools accept — the fight ends without any previously passing document turning red. `UPGRADING.md` documents this as the resolution of the previously unsatisfiable dual-gate state.

### format-frontmatter CLI hardening

The observable changes are the help text stating "default: check" and the skip diagnostics: `collect_paths` reports explicitly named files that exclusion drops, and the check/write loops report denylisted named files, each with a `skipped (…): PATH` stderr line; exit codes are unchanged. The dead `--check` flag is resolved through an internal mode-resolution unit with a stated contract, tested at unit level — its observable behavior is intentionally identical to the default, so no consumer-visible RED regression is claimed for it. The `docs/usage.md` reference gains the skip diagnostics and keeps the documented default.

### Legacy-authority note

The single message function keeps its condition, once-guard, and stderr channel; only the text changes to a factual statement of what the tool is doing (reading legacy authority read-only) rather than an instruction that contradicts `UPGRADING.md` §2. Tests cover the reworded note through all three call-site routes — resolution-routed commands, legacy-only validation, and the frontmatter compatibility wrapper — and the existing pinned assertion updates with it.

### Whole-file conflict diagnostics

`CP-CONSUMER-CONFLICT` findings for whole-file text targets additionally carry the first difference's location (line number) and a bounded excerpt of the expected package-side content only, rendered under the existing digest lines. Consumer-side content is never excerpted — terminal, JSON, and serialized findings stay digest-only for the actual bytes, per the control-plane confidentiality contract — and the pointer plus the consumer's own editor is sufficient for self-diagnosis. Binary or undecodable targets keep digest-only rendering. The finding schema change is additive and is exercised against secret-like consumer content to prove non-disclosure.

### Package and release integration

The three successor payloads are immutable full copies with updated integrity inventories. Family indexes and Catalog 5 retain old versions and advance only the consumer defaults; source payload projections, the rendered catalog, activation-test constants, and reconstruction tests refresh together, extending the parity proofs to the new versions. The engine fixes and successor packages ship together as Project Standards 5.8.0 following the released 5.7.0 process: one candidate wheel and sdist for all artifact-sensitive checks, release commit merged from `testing` to `main`, signed `v5.8.0` and moving `v5` tags, GitHub release assets, all release-commit workflows green, downloaded-asset byte parity, `testing` synchronized, and only then issue closure.

## Alternatives rejected

1. **Treat `testpaths` as a consumer-owned key (the issue's alternative).** Collection roots are load-bearing for the standard's gate semantics — an unmanaged `testpaths` could silently exclude the suite the coverage gate measures. A governed closed option keeps the value reconciled and conflict-diagnosable.
2. **`additional_test_roots` as an additive list.** The consumer need is replacing `["tests"]`, not extending it; an additive option cannot express "do not collect `tests/`", which is the reported repository's requirement.
3. **Parsed-JSON comparison for `.markdownlint.json` (the issue's second alternative).** A semantic-equality carve-out for one format weakens the digest-identity integrity primitive everywhere and invites format-specific comparison creep; the known-digest lineage already models "same content, different released bytes" and has the issue #10 precedent.
4. **Normalizing the shipped `.markdownlint.json` to literal UTF-8 instead.** That mutates a released payload's byte form for cosmetic reasons, invalidating every existing 1.7 lock digest — forbidden by payload immutability — and merely moves the mismatch to consumers holding the escaped form.
5. **Excluding frontmatter from Prettier (markdown-tooling change).** Prettier has no supported switch that skips frontmatter while formatting the body; faking it with ignore ranges would inject markup into consumer documents. It also leaves the doubled-apostrophe form, which is the less readable serialization.
6. **Documenting the quote conflict as a known deviation.** Two companion packages whose documented compliance checks cannot both be satisfied is a contract defect, not a caveat; the fix is mechanical and bounded.
7. **Making bare `format-frontmatter` an error (`one of --check or --write is required`).** The bare form has been documented check-mode since 3.0.0 and the symptom that motivated the error proposal did not reproduce; breaking every existing bare invocation to prevent a hypothetical misreading fails NFR-004.
8. **Normalizing all frontmatter scalars to the minimal-escape form, flagging the legacy single-quoted spelling.** A currently canonical `'Apple''s'` would newly report `would reformat`, a pass-to-fail check outcome that the previously-passing rule classifies MAJOR without exception — and the shared engine formatter would impose it even on exact 1.4 pins. Accepting both forms resolves the fight within MINOR.
9. **Suppressing the legacy warning only for `size-report`/`shape-check`.** A per-command suppression matrix leaves the contradiction for `validate` and future read-only commands and grows a second source of truth about command intent; one accurate sentence fixes every current and future emitter.
10. **Automatically deleting the legacy `validate` caller job during markdown-frontmatter migration.** Semantic identification of "our" job inside arbitrarily customized consumer YAML is guess-based writing in exactly the place the control plane promises fail-closed behavior; explicit ownership plus documented consumer maintenance is deterministic and lossless.
11. **Preserve-by-default bounded takeover for the unknown customized caller (`unknown_content_disposition = "preserve"` plus a provider managed-preserve claim).** The payload model forbids combining unknown-content preservation with a `consumer_owned_intent_pointer` on one signature, and dropping the pointer would remove the explicit relinquishment escape the issue asked for; silently composing into arbitrary unknown caller content would also leave a live legacy job behind by default. The declared-ownership branch keeps unknown content fail-closed with a followable escape, matching the sibling packages' accepted resolution.

## Failure behavior
