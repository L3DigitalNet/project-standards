# Combined Contract Audit — Ownership-Relinquishment Contract

## Executive summary

This audit reviews the proposed ownership-relinquishment contract — the narrow, not-yet-implemented exception that would let the migration engine resolve an _unrecognized whole-file_ legacy object as `consumer-owned`/`preserve` without the usual known-digest requirement. The contract is recorded across ADR 0023 (D1), SPEC-CP01 rev 0.9 (D2), SPEC-BA02 rev 0.10 (D3), the parallel-coverage design (D4), the migration research note (D5), and the D6 repo-state surfaces. Every finding below was verified against the engine's actual bytes — the two fail-closed gates in `src/project_standards/control_plane/migration.py`, the migration-report schema, and the shipping provider code — not against document prose alone. That grounding matters here specifically: the earlier round-4 audit declared convergence on the strength of the schema/`_safe_disposition` pairing and missed that gate #1 fires during signature inspection before ownership can matter and gate #2 rejects unknown digests. This review was scoped to catch exactly that class of text-plausible-but-engine-false error.

The defect diagnosis (DEF) and the fix mechanics (FIX) both hold up under byte-level scrutiny: the docs correctly describe the two-gate structure, correctly attribute the root cause to a generic engine ownership-transition gap, and specify a fix that opens both gates while leaving the "no create/update/remove/adopt/lock" guarantee intact. Cross-document coherence (COH) is substantively clean. The blocking concerns are concentrated in the adversarial-safety dimension (SEC) and in two document-internal consistency defects (DIG, TRC), plus a live-state repo-coherence disagreement (REF).

Seven findings survived verification: three at major severity (SAFETY-01, BC-01, REF-01), three at minor (TRACE-01, SAFETY-02, TRACE-02), and one nit (COH-01). The single most consequential is SAFETY-01: the exception's central safety promise — that a relinquishment is _target-specific_ — is asserted but not engine-verifiable as designed, because nothing binds the `intent_pointer` to the claimed target.

## Verdict

**The contract is NOT yet ready for the combined audit-and-implementation gate.** Blocking findings must be resolved in the governing documents first.

- **SAFETY-01 (major)** is the gating defect: as specified, the five enumerated engine proofs do not establish that the consumer's authorization pointer governs the specific file being relinquished, so a malicious or buggy provider can relinquish a _different_ file than the consumer authorized and suppress the fail-closed unknown-digest block for it. The exception cannot be implemented safely until target-specificity is made engine-verifiable (a declared pointer↔target binding, or a raw value that names the target).
- **BC-01 (major)** is a genuine internal contradiction that would break the stated backward-compat guarantee — and, if implemented literally, break an existing shipping known consumer-owned claim (cli-documentation). It must be corrected before implementation because a literal reading of DR-006/EC-013 diverges from the controlling FR-037.
- **REF-01 (major)** is a live-state coherence defect on the auto-injected surface an agent reads first; it points work at a premature migration and bypasses the audit gate itself. It should be corrected immediately, independent of the contract's implementation schedule.

The remaining findings (SAFETY-02, TRACE-01, TRACE-02, COH-01) are non-blocking hardening and editorial items but should be folded into the same revision pass. The underlying design is sound; the defects are in the _specification of the safety proofs and the traceability/status bookkeeping_, not in the engine understanding.

## Dimension coverage

**COH — Cross-document contract coherence.** Substantively clean. All eight canonical-contract points are stated consistently across D1–D5: whole-file-only eligibility with bounded blocks failing closed; explicit raw `consumer-owned` selection with a schema default held insufficient; the required claim quartet (`ownership="consumer-owned"`, `disposition="preserve"`, observed target+digest, `intent_pointer`) using value names that match the existing schema enums; the five engine proofs including "resolved payload materializes no artifact/contribution"; `intent_pointer` required-only-for-the-exception and forbidden on ordinary/known claims; preview path+digest exposed and labeled unvalidated; stale-plan apply binding; no create/update/remove/adopt/package-unit/lock action; `known_content_digests` retaining its "previously shipped" meaning with the observed digest as preview/apply evidence only; and the separate reviewed return-to-managed path. The D6 repo-state docs accurately carry the CP01 rev 0.9 / BA02 rev 0.10 numbers and the "pending combined audit / Not Started" status. The only divergence is a single enumeration nit in the ADR (COH-01).

**DEF — Technical soundness of the defect diagnosis.** Clean. The defect narratives in D4 and D5 match real engine behavior point for point: the optimized workflow digest `sha256:9f4f90…` is absent from every package signature (a whole-repo grep finds no match), and Python Tooling's `legacy-check-workflow` signature knows only the frozen V1 digest `sha256:16a65f…` (`payload.toml:662-666`); gate #1 fires inside `_inspect_signatures` whose findings are collected before any provider is invoked; gate #2 independently re-rejects unknown digests at `migration.py:914`; and `consumer-owned`/`preserve` over unknown bytes is therefore unreachable despite `_safe_disposition` permitting the pairing. The finding code, the ordering, and the root-cause attribution (a generic engine ownership-transition gap, not coverage.py or a provider bug) are all correct and neither overstated nor understated. The docs also correctly record that the round-4 prose-only audit missed this defect.

**FIX — Soundness and completeness of the proposed fix.** Clean. The change opens both fail-closed gates and the wiring is specified. D4:65 names each gate explicitly, directly correcting the round-4 blind spot; gate #1 deferral is stated in CP01/BA02 FR-037 and D4:112, and gate #2's relaxation is expressed as the acceptance clause "may resolve an unrecognized whole-file signature as consumer-owned only when…", with the deferred finding re-emitted for the non-exception case. The downstream engine paths were independently confirmed to cooperate: a `preserve`/`consumer-owned` claim skips `_adopted_legacy_units` (requires ADOPT), `_removal_actions` (requires REMOVE/IMPORT_LOCK), and the UNCLAIMED-ARTIFACT path (requires `item.known`), so the "no create/update/remove/adopt/lock" guarantee holds once the two gates are opened. The resolved-payload target-exclusion proof is adequately grounded in a config-conditioned `when_any` predicate. No internal contradiction rose to a defensible finding in this dimension.

**SEC — Fail-closed / abuse analysis of the exception.** Not clean. The fail-closed scaffolding for the standard blocked classes (bounded-block, managed/adopt, destructive/remove, shared, package-lock) is correct and consistent, and the anti-default guard (require a raw literal `consumer-owned` read from pre-schema legacy input) is sound. But the exception's target-specificity promise is asserted, not engine-verifiable: nothing binds the `intent_pointer` to the claimed target, so the five proofs reduce to "some recognized raw setting equals `consumer-owned` plus at most one claim per pointer," which permits misdirected relinquishment (SAFETY-01, major). Separately, moving the unknown-digest finding off the unconditional gate #1 lacks an explicit "still emit for every uncleared unknown whole-file signature" invariant, risking a fail-closed hole for unknown-and-unclaimed files (SAFETY-02, minor).

**TRC — Requirement traceability and spec completeness.** Not clean, though both specs are largely well-wired. In CP01, FR-037/FR-038/AW-012/EC-018/EC-019/ERR-013/R-009 are all defined and co-listed in the §17.3 matrix with the correct "Not Started — pending combined audit and implementation" status, tied to goals G-003/G-005, DoD item 847, and milestone MS-4; no orphan requirements or dangling rows exist. BA02 adds every required category and correctly reopens its Definition of Done. Two document-internal defects remain: a CP01 DoD checkbox still asserting full Must-completion despite two Not-Started Must requirements (TRACE-01), and a BA02 traceability-status gap where the amended DR-006 is marked Passing without a pending-audit trace, inconsistent with CP01's dual-listing of the same entity (TRACE-02).

**DIG — `known_content_digests` semantics and backward compatibility.** Not clean. The three central sub-claims hold: BA02 keeps `known_content_digests` meaning unchanged and explicitly separates the FR-037 owner-resolution path from it (L635, L637); it states the relinquishment claim's observed digest and `intent_pointer` are "preview/apply evidence only and never package-history authority" (L269, L637); and FR-037 (L232) forbids `intent_pointer` on known-history and other ownership/disposition combinations, consistent with the engine's existing `consumer-owned`→`{preserve}` pairing and gate #2's known-content pass path. But two normative rows — DR-006 (L269) and EC-013 (L746) — state the `intent_pointer` requirement without the "unrecognized/unknown whole-file" qualifier that FR-037 carries, so read literally they contradict FR-037's forbiddance and its backward-compat acceptance criterion (BC-01, major).

**REF — Reference integrity, indexes, and repo-state coherence.** Not clean. Index/version/link integrity is largely sound: D5 is correctly indexed with matching title and dates, README and specs-plans agree on CP01 rev 0.9 / BA02 rev 0.10 pending audit+implementation, TODO keeps the coverage task open and names the combined audit as the next gate, and D4's status line matches. One major defect: the two auto-injected live-state surfaces (`state.md` "Current focus" and the newest session-log row) both skip from "refresh release evidence" straight to "atomic v5 migration," omitting the combined audit that STATUS/TODO/specs-plans/README all present as the gating P0 v5 item (REF-01). The claimed STATUS.md size-warning bullet-removal was investigated and rejected as unsupported (REF-02).

## Findings

### SAFETY-01: `intent_pointer`→target binding is provider-asserted and engine-unverifiable, so the "single-target / for that target" authorization cannot be enforced and permits misdirected relinquishment

- **Severity:** Major
- **Verdict:** CONFIRMED
- **Documents:** D2 SPEC-CP01, D3 SPEC-BA02, D1 ADR 0023
- **Locations:** `docs/specs/2026-07-10-consumer-standards-control-plane-spec.md:285`, `:322`, `:654`; `docs/specs/2026-07-10-standard-bundle-authoring-v2-spec.md:232`, `:269`; `docs/adr/adr-0023-unified-consumer-standards-control-plane.md:133`; `src/project_standards/schemas/migration-report.schema.json:108`; `standards/python-tooling/versions/1.1/providers/python_tooling.py:610`; `src/project_standards/control_plane/migration.py:914`

- **Issue.** FR-037 (D2:285, D3:232) and AW-012/DR-006 require that the raw legacy input select consumer ownership "for the package target" / "the target's consumer-owned mode," and state the `intent_pointer` "may authorize exactly one target." But the only engine-checkable proofs the contract enumerates are: pointer ∈ provider `recognized_settings`; raw value literally `consumer-owned`; at most one claim per pointer; claim self-consistency (target exists and digest matches observed bytes); and payload materializes nothing for the target. None of these verifies that the pointer is the setting that _governs that specific target_. `recognized_settings` is a flat array of JSON-pointer strings with no target association, and the provider pairs an arbitrary pointer with an arbitrary target entirely in its own code. The engine has no declared pointer↔target map to check against. Consequence: if a consumer sets exactly one recognized setting to the literal `consumer-owned` intending to relinquish file A, a malicious or buggy provider can instead emit a single relinquishment claim for a different, sensitive file B (e.g. `.github/workflows/release.yml`) citing that same pointer. All five proofs pass; the unknown-digest fail-closed block that gate #1 / gate #2 would otherwise raise for B is suppressed; B is silently preserved and labeled "consumer-owned/unvalidated"; and the migration succeeds. The "single-target" guarantee as specified only prevents one pointer from relinquishing _many_ targets — it does not bind the pointer to the _correct_ target, re-introducing provider self-certification of _which_ file is relinquished.

- **Evidence.** `src/project_standards/schemas/migration-report.schema.json:108-115` defines `recognized_settings` as a bare array of strings with no target field; the package-level schema (`standards/python-tooling/versions/1.1/schemas/migration-report.schema.json:63`) is identical. LegacyClaim (schema:41-75) carries `target`+`observed_digest` but no pointer linkage. `migration.py:826-887` (`_coverage_findings`) treats `recognized_settings` purely as flat pointers checked for presence/overlap, never mapping a pointer to a target. `standards/python-tooling/versions/1.1/providers/python_tooling.py:610-643` shows `run_migrate` building `recognized` from legacy config keys and `claims` with target chosen per `_SIGNATURES` in separate, independent loops, so the pointer↔target pairing lives entirely in provider code. Gate #1 (`migration.py:688-710`) and gate #2 (`migration.py:914`) are the fail-closed unknown-digest emitters the exception is designed to bypass. The contract text (D2:285, D3:232, D1:133) asserts target-specific/single-target authorization but enumerates no proof establishing the pointer→target correspondence. "May authorize exactly one target" is, per FR-037's own acceptance criteria, the one-claim-per-pointer reuse check — not a declared binding. Severity is bounded by the trusted-provider model and a narrow silent-harm window (the misdirected target must be a present unknown-digest whole file; the intended target, if present, would still trip gate #1), but the replacement is strictly _less_ target-bound than the known-digest gate it removes.

- **Recommendation.** Make target-specificity engine-verifiable rather than provider-asserted. Add a declared, static, offline-auditable pointer↔target mapping to the payload manifest (each relinquishable target names the exact recognized setting that authorizes it) and require the engine to prove `claim.target == declared_target(intent_pointer)`. Alternatively, require the raw legacy value at the pointer to itself name the target path and verify it equals `claim.target`, so the consumer's authorization is intrinsically bound to the file. Until such a binding is specified, downgrade the "authorizes exactly one target" language to what is actually enforceable and document the gap. Update FR-037 (D2/D3), AW-012, DR-006, and the ADR relinquishment paragraph accordingly.

### BC-01: DR-006 and EC-013 require/constrain `intent_pointer` on ALL consumer-owned preserve claims, contradicting FR-037's known-history carve-out and the backward-compat guarantee

- **Severity:** Major
- **Verdict:** CONFIRMED
- **Documents:** D3 SPEC-BA02
- **Locations:** `docs/specs/2026-07-10-standard-bundle-authoring-v2-spec.md:269`, `:746`, `:232`, `:637`; `src/project_standards/control_plane/migration.py:184`, `:914`

- **Issue.** FR-037 (L232) scopes `intent_pointer` to the unknown-whole-file exception and states it "is forbidden when the claim matches declared package history or uses another ownership/disposition pair," with the explicit Must acceptance criterion "preserves existing known consumer-owned claims without the field." But two other normative rows drop the "unrecognized/unknown whole-file" qualifier and speak of `consumer-owned`/`preserve` claims unconditionally: DR-006 (L269) says "a consumer-owned preserve claim … names a recognized raw `intent_pointer` whose value is `consumer-owned`," and EC-013 (L746) rejects any "consumer-owned preserve claim [that] … omits `intent_pointer`" (and any that "targets a bounded block"). Read literally, DR-006 _requires_ and EC-013 _rejects-for-lack-of_ the field on every consumer-owned/preserve claim — including an ordinary _known-content_ consumer-owned/preserve claim, which FR-037 requires to remain valid _without_ the field. This is an internal contradiction that, if implemented per DR-006/EC-013, breaks the very backward-compat guarantee the amendment promises. EC-013's blanket "targets a bounded block → reject" additionally over-restricts a known consumer-owned/preserve claim on a bounded-block signature, which the schema and `_safe_disposition` currently permit.

- **Evidence.** The engine confirms `consumer-owned`/`preserve` is a today-legal, field-free pairing for known content: `_safe_disposition` (`migration.py:184`) maps `consumer-owned` → `frozenset({PRESERVE})`; gate #2 `_claim_findings` (`migration.py:914`) passes a claim with no finding when `item.known` is true and the digest matches; and the current LegacyClaim model has `additionalProperties:false` with required fields `signature_id`/`target`/`observed_digest`/`ownership`/`disposition` and _no_ `intent_pointer`. A real shipping provider already emits exactly such a claim: `standards/cli-documentation/versions/1.1/providers/cli_documentation.py:220-224` returns `ownership="consumer-owned"` for `.github/workflows/cli-docs-check.yml` via `_known_claim` (L163-188), which sets `disposition="preserve"` and only fires when `observed.get("known")` is true — i.e. a known consumer-owned/preserve claim with no `intent_pointer`. This is precisely the "existing known consumer-owned claim without the field" that DR-006 would render non-conformant and EC-013 would reject, breaking cli-documentation's migration and the stated backward-compat guarantee. FR-037 (L232) and L637 confine `intent_pointer` to the "unrecognized whole-file" path and say the exception "does not recognize, adopt, import, validate, or extend package history"; DR-006 (L269) and EC-013 (L746) omit that confinement, so their text applies to known claims too. Severity is major but not blocker: FR-037 is the correctly-scoped controlling requirement and the contract is not-yet-implemented, so a careful implementer cross-referencing it could avoid the error — but as written the two rows are genuinely contradictory, target a Must backward-compat criterion, and would break an existing shipping known claim if implemented literally.

- **Recommendation.** Re-scope DR-006 and EC-013 to the FR-037 exception only. In DR-006 (L269), change "a consumer-owned preserve claim … names a recognized raw `intent_pointer`" to "an _unrecognized_ whole-file consumer-owned preserve claim (one whose observed digest is not in `known_content_digests`) echoes … and names a recognized raw `intent_pointer`; a known-history consumer-owned preserve claim carries no `intent_pointer`." In EC-013 (L746), qualify the trigger to fire only on the unknown-whole-file relinquishment path — e.g. "An unrecognized-whole-file relinquishment claim that omits `intent_pointer`, targets a bounded block, …" — so a known consumer-owned/preserve claim (including a known bounded-block one) that legitimately omits the field is not rejected. This aligns both rows with FR-037 and the backward-compat criterion without changing fail-closed behavior for the genuine unknown-signature case.

### SAFETY-02: Deferring the unknown-digest finding from unconditional gate #1 to claim-driven validation lacks an explicit "still emit for every uncleared unknown whole-file signature" invariant

- **Severity:** Minor
- **Verdict:** PLAUSIBLE
- **Documents:** D2 SPEC-CP01, D3 SPEC-BA02
- **Locations:** `docs/specs/2026-07-10-consumer-standards-control-plane-spec.md:285`, `:677`, `:718`; `docs/specs/2026-07-10-standard-bundle-authoring-v2-spec.md:232`, `:783`; `src/project_standards/control_plane/migration.py:699`, `:926`

- **Issue.** FR-037 (D2:285, D3:232) directs the engine to "defer the unknown-digest finding until claim validation." Today that finding is raised unconditionally at gate #1 (`migration.py:699`) for _every_ observed signature whose digest is not in `known_content_digests`, independent of whether any claim references it. Claim validation is claim-driven: `_claim_findings` (`migration.py:890-937`) iterates `report.claims` and, in its only non-claim branch (926-936), emits solely for items where `item.known and key not in claimed` — it never emits for an item that is both _unknown and unclaimed_. If the deferral is implemented by suppressing the gate #1 emission and relying on claim validation, an unknown whole file that the provider simply never mentions would pass with no finding — a fail-closed regression. EC-018/ERR-013 (D2:677,718) and ERR-011 (D3:783) guarantee the finding is retained only when a relinquishment claim is _rejected_; they do not state the invariant for the no-claim case.

- **Evidence.** `migration.py:688-710` shows gate #1 emits `CP-MIGRATION-LEGACY-DIGEST` for every `not known` observed signature during inspection, unconditionally. `migration.py:926-936` shows the only observed-item-driven branch in `_claim_findings` fires exclusively for known-and-unclaimed items (`if item.known and key not in claimed`); unknown-and-unclaimed items have no emitter there. A grep for unclaimed/no-claim/every-observed/uncleared language found no explicit invariant covering the unknown-and-unclaimed case. Marked PLAUSIBLE rather than CONFIRMED because "defer" semantically entails emit-unless-cleared (so the invariant is arguably already implied) and ERR-007 (D2:712) independently requires the report to surface "unresolved ownership," a general net over an unknown unclaimed file. It is a spec-explicitness/hardening ask, not a hard contradiction — hence minor.

- **Recommendation.** State the fail-closed completeness invariant explicitly in FR-037 and the relevant EC/ERR rows: every observed unknown whole-file signature not cleared by a fully valid FR-037 relinquishment claim MUST still emit `CP-MIGRATION-LEGACY-DIGEST`, and any observed unknown signature with no claim at all MUST emit it. Frame deferral as "hold-and-emit-unless-cleared," and add a test row covering the unknown-and-unclaimed (provider-omits-the-file) case, not just the rejected-claim case.

### TRACE-02: BA02 amended DR-006 traced as "Passing" with no pending-audit trace, unlike CP01's dual-listing

- **Severity:** Minor
- **Verdict:** CONFIRMED

### COH-01: ADR 0023 omits "observed target" from the required preserve-claim return fields that CP01/BA02 mandate

- **Severity:** Nit
- **Verdict:** CONFIRMED
- **Documents:** D1 ADR 0023, D2 SPEC-CP01, D3 SPEC-BA02
- **Locations:** `docs/adr/adr-0023-unified-consumer-standards-control-plane.md:133`; `docs/specs/2026-07-10-consumer-standards-control-plane-spec.md:285`; `docs/specs/2026-07-10-standard-bundle-authoring-v2-spec.md:232`; `src/project_standards/control_plane/migration.py:914`

- **Issue.** ADR 0023's Migration section enumerates the fields the migration provider must return for a preserve claim as `ownership="consumer-owned"`, `disposition="preserve"`, "the exact observed digest", and an `intent_pointer` (L133) — it names the digest but drops the observed _target_ from that required-field list. Both controlling specs require the target: CP01 FR-037 states the provider returns "the exact observed target and digest" (L285) and BA02 FR-037 likewise (L232). The target is load-bearing because engine gate #2 keys the claim-to-observation match on `(standard_id, signature_id, target.original)` and rejects a mismatch (`migration.py:898,914`), so a claim missing an exact target could never be validated.

- **Evidence.** ADR L133's required-return list reads "…the migration provider must return `ownership="consumer-owned"`, `disposition="preserve"`, the exact observed digest, and an `intent_pointer`…" — no `target`. CP01 L285 and BA02 L232 both read "…with the exact observed target and digest plus an `intent_pointer`…". The same ADR paragraph does reference the target elsewhere ("materialize no artifact or contribution for the target"; "exposes the preserved path and digest"; apply "bound to the observed bytes and file identity"), so the model is present but the enumerated claim contract is incomplete relative to the specs — hence a nit, not a substantive contradiction.

- **Recommendation.** Amend ADR 0023 L133 to read "…the exact observed target and digest, and an `intent_pointer`…" so the ADR's enumerated required-return fields match CP01 FR-037 and BA02 FR-037. Editorial alignment only; no behavioral change.
