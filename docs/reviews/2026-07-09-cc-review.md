## 🔴 Critical — will fail during TDD execution

### C1 · Task 3: duplicate-id test can never match, and the branch it targets is dead code

The plan adds a duplicate-id guard in `build_graph` raising `ValueError("duplicate standard id …")` and a test (`test_build_graph_rejects_duplicate_standard_ids`) that rewrites `standards/other/standard.toml` to `id = "alpha"` and asserts `pytest.raises(ValueError, match="duplicate standard id")`.

But the **Step 03 loader enforces `id == bundle_dir.name` and fires first** (`standard_manifest.py:324`). Reproduced:

```text
StandardManifestError message: standard id 'alpha' != bundle directory 'other'
is ValueError? True
matches "duplicate standard id"? False
```

`StandardManifestError` _is_ a `ValueError`, so `pytest.raises(ValueError)` catches it — but `match="duplicate standard id"` fails against the real message → **test fails**. Worse, the guarded branch is **unreachable**: every loaded node's id equals its directory name, and directory names under `standards/` are unique, so `seen` can never collide. That is dead code (and an uncovered line that eats into the Task 12 ≥85% coverage gate).

**Fix:** Drop the `build_graph` duplicate-id branch and its test. The real failure mode ("a bundle dir whose manifest id disagrees") is already the loader's job and already covered by Step 03. If a graph-level guard is still wanted for defense-in-depth, exercise it by constructing `StandardsGraph` with two `StandardNode`s sharing an id directly, not through `build_graph`, and assert the actual message.

### C2 · Task 9: `main(["--help"])` raises `SystemExit`, it does not return `0`

`test_top_level_help_advertises_standards` asserts `main(["--help"]) == 0`. The existing top-level CLI does **not** early-dispatch bare `--help`; it reaches `argparse.parse_args(["--help"])`, which prints help and calls `sys.exit(0)`. `main` has no `SystemExit` guard. Reproduced:

```text
usage: project-standards [-h] [--version] {validate,fix,spec,adopt,list} ...
RAISED SystemExit code= 0
```

pytest surfaces the `SystemExit` as a test error, not a pass. (Note the existing suite only ever calls `main(["spec","--help"])` / `main(["validate","--help"])`, both of which _are_ early-dispatched and return 0 — so there is no precedent for the bare form the plan assumes.)

**Fix:** Change the assertion to `with pytest.raises(SystemExit): main(["--help"])` and capture stdout to assert `"standards"` appears — or, if a clean `0` return is desired, add `standards` to a help string and keep the assertion off the argparse `SystemExit` path. Either way the plan's current line is wrong.

### C3 · Task 7: the hidden-dependency test asserts the wrong error and tests the wrong thing

`test_requires_field_is_rejected_as_hidden_dependency` calls `write_standard(..., extra_toml='\nrequires = ["beta"]\n')` and asserts `StandardManifestError` with `match="Extra inputs are not permitted"`.

The fixture writer appends `extra_toml` **after the `[resources]` table** (Task 2 helper, `text += extra_toml` is the last line). In TOML, a bare `requires = [...]` after `[resources]` belongs to the **resources table**, not the document root. `ResourcesTable` uses `extra="allow"` with typed-`str` extras, so this produces a _type_ error, not an _extra-forbidden_ error. Reproduced:

```text
>>> 'requires' landed under: resources
resources.requires
  Input should be a valid string [type=string_type, input_value=['beta'], input_type=list]
contains "Extra inputs are not permitted"? False
```

So `match=` fails → **test fails**. And the deeper problem: the test never exercises the guard it claims to (`_Table` `extra="forbid"` rejecting a top-level or `[relations]` `requires` key), so **FR-021's headline hidden-dependency case is effectively untested** despite Task 7 existing precisely to pin it.

**Fix:** Make the fixture place `requires` where it is actually rejected as an extra. Two options: (a) inject it under `[relations]` (e.g. an `extra_toml` that opens `[relations]`… no — `relations` is already emitted; instead give the helper a way to add a stray key to `[relations]` or `[standard]`), or (b) target the top-level document by writing a raw manifest that puts `requires` before the first table. Then assert `match="Extra inputs are not permitted"`. Also add a positive assertion that the loader message names `requires`.

> Note: Task 8's `test_validate_graph_exit2_on_load_error` reuses the same `requires` fixture but only asserts `rc == 2` / `code == "graph_load_error"` — it does **not** match the message, so it passes regardless. Only Task 7's `match=` is broken. That divergence is itself a smell worth fixing together.

## 🟡 Important — soundness gaps (compile/run, but weaken the guarantee)

### M1 · `targets_may_overlap` under-detects real authority conflicts, and its non-trivial branches are untested

The helper only flags overlap on: identical targets, `**/*`/`**/*.*` wildcards, `fnmatch` either-direction, or same recursive-`**/*`-suffix. It **misses** genuine overlaps such as `pyproject.toml` vs `**/*.toml` (`fnmatch` needs a literal `/`; the suffix helper only fires when both sides start `**/*`). For a validation **gate**, false negatives are the dangerous direction — two mutating authorities that really collide would pass `validate-graph` clean. This directly limits FR-004's completeness.

Compounding it: every authority test in the plan uses identical `**/*.md` targets, so only the `left == right` branch is ever executed. The `fnmatch` and suffix branches ship untested (behavior unverified + Task 12 coverage risk).

**Fix:** Either (a) document explicitly in the plan/spec that overlap detection is intentionally conservative and lists the accepted false-negative class (and add a test asserting a known-missed case is _not_ flagged, to make the limitation a decision rather than an accident), or (b) strengthen the check (e.g. normalize `**/*.ext` ⊇ `name.ext`). Add tests that exercise the suffix and `fnmatch` branches regardless.

### M2 · `GraphFinding(order=True)` + `standard_id: str | None` + `sorted()` = latent crash

`validate_graph` and `findings_to_jsonable` both `sorted(findings)`. `GraphFinding` is `order=True` with `standard_id: str | None`, and `_finding(node=None)` is an explicitly supported code path. The moment any None-owner finding coexists with a str-owner finding, sorting throws. Reproduced:

```text
TypeError on sort: '<' not supported between instances of 'NoneType' and 'str'
```

Today no call site passes `node=None` (missing-manifest findings use `path.name`), so it doesn't fire — but the type signature and helper invite it, and the first future None-owner finding will crash the CLI at exit rather than report. This is a landmine, not a live bug.

**Fix:** Either drop the `None` option from `standard_id` (use `""`), or give the dataclass a `sort_key` that coerces `None`, or sort with an explicit `key=` that tolerates `None`. Add a test with a `None`-owner finding in the mix.

### M3 · `_has_adr_resource` is too loose to actually enforce "extends must be ADR-backed" (FR-021)

The check is **per-node, not per-edge**, and matches any resource id in `{adr, extension_adr}`, any id starting `adr_`, or any value under `resources/adr*` / `resources/extension-adr*`. Consequences: a standard extending _multiple_ bases satisfies the rule with a single ADR; a standard can satisfy it with an unrelated `adr_notes` resource that documents nothing about the extension; the ADR is never tied to the specific `extends` target. The plan's Global Constraints promise "reject extension relationships that … lack an ADR resource" — this passes shapes that meet the letter but not the intent.

**Fix:** At minimum require the resource id to be exactly `extension_adr` (the bundle-authoring convention) rather than a fuzzy prefix match, and state in the plan that per-edge ADR attribution is out of scope for Step 04 (if it is). Otherwise an executor will ship the loose heuristic believing it enforces the constraint.

## 🔴 New blockers introduced by the fixes

### C4 · The M2 fix was applied to `model.py` but not to `validators.py` — `validate_graph` now crashes on ≥2 findings

Removing `order=True` from `GraphFinding` was correct, and the two renderers in `model.py` were updated to `sorted(findings, key=_finding_sort_key)`. But `validate_graph` still ends with a **bare** `return sorted(findings)` (plan line 822). With no `__lt__` on the dataclass, this raises:

```text
TypeError: '<' not supported between instances of 'GraphFinding' and 'GraphFinding'
```

(reproduced with the exact post-fix dataclass definition). This fires the moment two findings coexist — which is the **common** path, not an edge case:

- `test_duplicate_config_namespace_is_error` (Task 4) expects exactly two findings → `validate_graph` crashes → **Task 4 Step 4 fails**.
- Cascades to every multi-finding validator/CLI test and to `test_current_repo_validate_graph_require_all_manifests_reports_step05_gap` (7 missing dirs → 7 findings → crash).

Note the regression test that _was_ added for M2 (`test_ownerless_findings_sort_without_type_error`) only exercises `findings_to_jsonable` — the layer that got the key. It does **not** cover the `validators.py` sort, so it gives false confidence. This is strictly worse than the original M2 latent bug: it's now an active crash on the default path.

**Fix:** give `validate_graph` the same key. Cleanest is one shared sort helper: export `finding_sort_key` (or a `sort_findings()` function) from `model.py` and use it in all three places — `validate_graph`, `findings_to_jsonable`, `format_findings` — so there is a single ordering definition. Then extend `test_ownerless_findings_sort_without_type_error` (or add a validator-level test) to call `validate_graph` on a ≥2-finding graph so this can't regress silently again.

### C5 · `targets_may_overlap` is imported in the Task 4 test header but not implemented until Task 5 → ImportError at the Task 4 checkpoint

The M1 fix added `targets_may_overlap` to the Task 4 test-file import (plan line 649):

```python
from project_standards.standards_graph.validators import targets_may_overlap, validate_graph
```

But `targets_may_overlap` is defined in **Task 5** (plan line 1018); Task 4's `validators.py` body contains it **0 times** (confirmed). Python resolves the import at module-collection time, so when Task 4 Step 4 runs `pytest tests/test_standards_graph_validators.py`, collection dies with `ImportError: cannot import name 'targets_may_overlap'` — every Task 4 validator test errors instead of the plan's expected PASS.

**Fix (pick one):**

- Move `targets_may_overlap` (and its helpers) into the Task 4 `validators.py` implementation, since the symbol is now referenced from the Task 4 test module; or
- Keep the function in Task 5 but add `targets_may_overlap` to the import line **in Task 5's test-append step**, leaving Task 4's header importing only `validate_graph`; or
- Import it lazily inside the Task 5 test functions.

The second option preserves the current task boundaries with the smallest diff.

## 🟡 Remaining blocker

### M5 · Mid-file imports in the test files fail Ruff E402 at the close gate

The repo enforces `E402` (`[tool.ruff.lint] select = ["E", …]`, only `E501` ignored) and `[tool.ruff.lint.per-file-ignores]` waives just `S101` for `tests/**/*.py` — **E402 applies to tests**. Confirmed with a probe under the real config: an import placed after a `def` in a `tests/` file raises `E402 Module level import not at top of file`, and E402 is **not** in Ruff's autofixable set.

Five appended imports land after function definitions across four tasks:

| Plan line | Task | Appended import                                               |
| --------- | ---- | ------------------------------------------------------------- |
| 507       | 3    | `from …standards_graph.discovery import (build_graph, …)`     |
| 893       | 5    | `from …standards_graph.validators import targets_may_overlap` |
| 1354      | 7    | `import pytest`                                               |
| 1356      | 7    | `from …standard_manifest import StandardManifestError`        |
| 1602      | 9    | `from project_standards.cli import main`                      |

Each per-task `pytest` checkpoint still passes (pytest doesn't run Ruff), so the failure stays invisible until **Task 12 Step 2** (`uv run ruff check .`), where all five surface at once and block the close gate. (The C5 fix added the line-893 one; the other four predate it — I missed them in rounds 1–2. The `__init__.py` "append exports" appends are **fine**: Ruff exempts `__all__` dunder assignments from E402, verified.)

**Fix:** In each "append to tests/…" step that introduces a new symbol, add the import to the **top import block** of that file rather than inline — then append only the new test function(s) below. This still respects the per-task symbol-availability constraint (the import is introduced in the same task the symbol becomes importable, e.g. `build_graph` at Task 3, `targets_may_overlap` at Task 5), so no checkpoint imports a not-yet-existing name. Concretely, reword those five steps to "add `<import>` to the file's import header, then append:" and move the import line out of the appended block.

Optional belt-and-suspenders: keep `ruff check` off the per-task loop but add a note to Task 12 that E402 in the new test files means an import was appended inline and must be hoisted.
