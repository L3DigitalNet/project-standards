# Housekeeping and Repo-Hygiene Workflow

Recurring maintenance tasks that keep a repository healthy. They are not tied to a release or milestone; run the full list periodically — at minimum alongside each release, or when drift is noticed. Do not check items off in this file: the list is a template, re-run each time.

## Locate the Targets

Before working through the checklist, identify from the repository's own conventions (agent instructions, documentation index, directory layout):

- Where ephemeral documents live: plans, reviews, scratch/working files.
- Which documents are managed (governed by frontmatter or metadata standards) and which config defines that scope.
- Which indexes, inventories, and catalogs the repository maintains, and whether any are generated by a tool.

Do not assume locations from memory — repository layouts change; the repository's current config and instructions are authoritative.

## Cleanup Loose Files

Loose files accumulate outside of their intended locations. Identify and move them to the correct location. Loose files may include:

### Research

Research belongs in `docs/research/`. This includes durable reports generated at the user's request, by a skill, or through a similar workflow. Specs, plans, and reviews keep their dedicated locations.

### Scratch

Scratch material belongs in `docs/scratch/` unless a standard, ADR, or convention governs its location. Add `docs/scratch/` to `.gitignore`. If the directory is absent, create it with a single `scratch.md` placeholder.

## File Deletion

- [ ] Delete completed plans. Plans are ephemeral; specs are the historical record.
- [ ] Delete completed reviews. Reviews are ephemeral; once findings are resolved, the target document is the record.
- [ ] Move loose scratch files to `docs/scratch/`. Do not delete files in `docs/scratch/` unless they are genuinely stale or irrelevant and were not user-authored.
- [ ] Identify and delete stale specs.
  - Stale specs are those that have been superseded by a newer spec, or that are no longer relevant to the repository's current state.
  - Before deleting a spec, ensure that it is not referenced by any other document or index in the repository.
  - If a spec is still relevant but has been superseded, consider archiving it instead of deleting it.
- [ ] Before deleting anything, confirm it is genuinely complete or stale — check the work queue, status docs, and recent history. Never delete user-authored content without confirmation.

## Frontmatter and Metadata

For every managed document in the repository's governed scope:

- [ ] State-dependent fields (e.g. `status`, `confidence`, `updated`, `reviewed`) reflect the document's current state.
- [ ] Relationship fields (e.g. `related`, `supersedes`, `superseded_by`) match the actual relationships: add new links, remove stale ones.
- [ ] Classification fields (e.g. `tags`, `aliases`) match the current content.
- [ ] `description` (or equivalent summary field) still describes the document; rewrite it if the content has shifted.
- [ ] Metadata complies with whichever standards, ADRs, and conventions govern it in this repository.

## Indexes and Inventories

- [ ] Every index and inventory matches the files on disk — no dangling rows, no missing entries.
- [ ] Generated indexes and catalogs are regenerated (or their check command passes) rather than hand-edited.
- [ ] Items removed from the repository are also removed from every index that referenced them.

## Links and References

- [ ] Relative links in tracked documents resolve to existing files (no broken paths after moves or deletions).
- [ ] References to renamed, superseded, or deleted documents are updated or removed.

## Validation and Closeout

- [ ] Run the repository's documented validation gates for documentation (frontmatter/metadata validators, formatters, linters) and confirm they pass.
- [ ] Record the completed pass wherever the repository tracks status and session history — not by checking boxes here.
